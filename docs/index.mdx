A small library of **fused Triton kernels** — each with both a forward pass and a custom autograd
backward — that drop in for an eager PyTorch block during transformer training. They are developed and
tuned on the Turing / Tesla T4 class (`sm_75`), and correctness is always the gate: numerical parity
of both the forward output and the gradients against the eager reference is never traded for speed.

## What this is — and isn't

This is **not** a "compile everything" layer. `torch.compile` already saturates HBM bandwidth on plain
elementwise ops; you cannot beat it there, and this library does not try. Instead, every kernel here
targets a place where the compiler *structurally* cannot fuse — a routing decision whose shape is only
known at runtime, a loss that would have to materialize an enormous intermediate, a reduction the
compiler insists on doing in two passes, or a native op it must keep as an opaque library call.

If you want the one idea to take away, read [The structural edge](/concepts/the-structural-edge).

## The kernels

  <Card title="Fused-linear cross-entropy" icon="layer-group" href="/kernels/cross-entropy">
    Fuses the LM-head GEMM with the softmax-CE so the `(N, V)` logits are never materialized. Trains
    where standard cross-entropy OOMs; matches Liger's fused CE.
  </Card>
  <Card title="XSA correction" icon="arrows-to-dot" href="/kernels/xsa">
    Exclusive Self-Attention rejection fused into one forward and one backward kernel that reads `V`
    only once, with in-kernel GQA broadcast.
  </Card>
  <Card title="Fused Muon" icon="gauge-high" href="/kernels/muon">
    Polar-Express Newton-Schulz optimizer step, launch-overhead-fused with `foreach` + `baddbmm` and a
    bit-identical distributed variant for DDP.
  </Card>
  <Card title="Conv MoE router" icon="route" href="/kernels/router">
    Conv → sigmoid gate → top-k → gather, fused into one epilogue (folding native `topk`) with a merged
    manual backward.
  </Card>
  <Card title="PolyGLU MoE combine" icon="diagram-project" href="/kernels/moe">
    Data-dependent expert dispatch + cuBLAS GEMMs + fused activation + weighted scatter — the routing
    edge the compiler can't fuse.
  </Card>
</Columns>

## Performance at a glance

All numbers are measured on a **Tesla T4 (`sm_75`)** against a `torch.compile`'d eager baseline.

| Kernel | T4 result vs `torch.compile` |
|---|---|
| `fused_linear_cross_entropy` | ~3.4× less peak memory (trains where standard CE OOMs); matches Liger |
| `fused_xsa` | ~1.15× forward+backward, grad-exact |
| `FusedMuon` | ~1.09× (75M params) / ~1.05× (300M), peak memory ≤ baseline |
| `fused_router` | ~1.11–1.17× forward+backward, exact grads, memory parity |
| `moe` | ~2.87× (the data-dependent routing edge survives compile) |

<Warning>
Kernel performance is **architecture-specific**. These numbers were obtained on Turing (`sm_75`) and do
not transfer to other GPU classes. Always re-benchmark on your own target hardware — see
[Benchmarking](/concepts/benchmarking).
</Warning>
