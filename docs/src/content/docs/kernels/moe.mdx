The MoE **combine** step: given a router's top-k indices and weights and a set of stacked expert weights,
dispatch each token to its experts, run the per-expert GEMMs and activation, and scatter the weighted
results back. The data-dependent dispatch is something `torch.compile` cannot fuse at all, which is why
this is the largest win in the library.

<Aside type="note">
This kernel is shaped for the **PolyGLU** expert layout used in BiBo — a fused gate+up projection with
per-expert activation codes. The per-expert dispatch machinery is general, but the activation set and the
weight layout are opinionated. Adapt the `act_codes` and weight shapes to your model.
</Aside>

## Why it wins

In a Mixture-of-Experts layer, the router decides which tokens go to which expert **at runtime**. The
compiler plans a static graph, so it cannot fuse the dispatch, the per-expert cuBLAS GEMMs, the PolyGLU
activation, and the weighted scatter into one data-dependent pass — it falls back to a masked loop with a
GPU sync per expert and an unfused activation. This kernel does the whole thing in one fused path. It is the
canonical data-dependent routing edge.

## Performance

Measured on a Tesla T4 (`sm_75`):

- **~2.87×** vs the compiled masked per-expert combine — and this is the win that survives the compiled
  baseline by the widest margin, because the routing simply cannot be compiled.

**On Blackwell (`sm_120`)** the per-expert path widens to **~3.9× forward+backward** (backward ~4.6×),
correct and lighter on memory — the data-dependent-routing edge scales hard with the faster tensor cores.
See [Hardware and portability](/triton-kernel-fused/concepts/hardware/).

Blackwell also unlocks a second path. When a model has **many experts** (real-MoE sparsity), the
per-expert loop becomes a tail of tiny GEMMs and `sm_120` instead routes to a single cuBLAS *grouped*
GEMM — `moe_grouped_cublas_polyglu` — that beats per-expert by **~2–3× at or below its memory**. `moe()`
picks it automatically; see [Grouped path on Blackwell](#grouped-path-on-blackwell-sm_120).

## API

```python
moe(
    hidden,            # (N, H) — token hidden states
    top_k_indices,     # (N, k) int — expert ids from YOUR router
    top_k_weights,     # (N, k)     — routing weights from YOUR router
    gate_up_proj,      # (E, 2*I, H) — fused gate+up projection per expert
    down_proj,         # (E, H, I)   — down projection per expert
    act_codes,         # (E,) int32  — per-expert kind: 0=SiLU 1=ReLU² 2=Tanh (GLU); 3=Identity 4=Zero
)  # -> (N, H) combined output
```

| Argument | Shape / type | Notes |
|---|---|---|
| `hidden` | `(N, H)` | Flattened token hidden states. |
| `top_k_indices` | `(N, k)` int | Chosen expert ids — comes from your router (e.g. [`fused_router`](/triton-kernel-fused/kernels/router/)). |
| `top_k_weights` | `(N, k)` | Routing weights for the chosen experts. |
| `gate_up_proj` | `(E, 2·I, H)` | Stacked fused gate+up projection, one slot per expert (`E` counts special experts too — see below). |
| `down_proj` | `(E, H, I)` | Stacked down projection, one slot per expert. |
| `act_codes` | `(E,)` int32 | Per-expert **kind**: `0`=SiLU, `1`=ReLU², `2`=Tanh (GLU experts); `3`=Identity, `4`=Zero (special experts). |

The router stays in **your** model — this kernel only takes its top-k output and the stacked expert weights.

### Special experts (Identity and Zero)

`act_codes` selects each expert's **kind**, not just its activation. Beyond the three GLU activations, two
codes pick non-GLU experts — the MiMo / BiBo "9 GLU + Identity + Zero" stack uses both:

| `act_codes[e]` | Expert kind | Computes | Uses its weight slots? |
|---|---|---|---|
| `0` | SiLU GLU | `down(SiLU(gate) · up)` | yes |
| `1` | ReLU² GLU | `down(ReLU²(gate) · up)` | yes |
| `2` | Tanh GLU | `down(Tanh(gate) · up)` | yes |
| `3` | **Identity** | weighted passthrough: `out += w · hidden` | **no** (slot ignored) |
| `4` | **Zero** | nothing — the token is dropped for that expert | **no** (slot ignored) |

There is no separate API for special experts: you just set their code in `act_codes`. Two rules:

- **`E` (and the first dim of `gate_up_proj` / `down_proj`) counts the special experts.** Their weight slots
  are **never read**, but they must exist so expert id `e` keeps indexing the stacked weights directly —
  fill them with zeros (or any placeholder). For the 11-expert BiBo stack, `gate_up_proj` is `(11, 2I, H)`
  with two unused rows.
- **`moe` routes correctly automatically.** A code `> 2` means a special expert is present, so `moe` uses
  the per-expert path (the grouped path is GLU-only and would compute wrong math on codes `3`/`4`). Identity
  and Zero are handled in both forward and backward — Identity passes the routing-weight gradient through
  (`dx = w · go`, `dw = Σ go · hidden`); Zero produces no gradient.

```python
# 9 GLU experts (SiLU/ReLU²/Tanh triples) + Identity + Zero -> 11 experts total
act_codes = torch.tensor([0, 1, 2, 0, 1, 2, 0, 1, 2, 3, 4], device="cuda", dtype=torch.int32)
# gate_up_proj / down_proj are sized (11, ...); slots 9 (Identity) and 10 (Zero) are placeholders.
```

## Grouped path on Blackwell (`sm_120`)

The per-expert path fires one cuBLAS GEMM **per expert**. That is ideal when each expert receives many
tokens — its GEMMs are large and saturate the tensor cores. But real MoEs route a few tokens to many
experts, so the per-expert loop degenerates into a long tail of tiny, tensor-core-starved GEMMs, each
wrapped in a Python iteration with a device-to-device copy (profiled at ~1313 CUDA launches / ~510 DtoD
copies per step with 32 experts).

`kernels/sm120` adds **`moe_grouped_cublas_polyglu`**: one *grouped* GEMM over all routed tokens via
[`torch._grouped_mm`](https://pytorch.org/docs/stable/generated/torch._grouped_mm.html) — a cuBLAS grouped
GEMM that is **autograd-native** in bf16/fp16 — plus the PolyGLU activation and a weighted scatter. That
collapses the loop to a few scheduled GEMMs (~851 launches / ~35 copies). It is **correct on the
special-experts stack**: the GLU experts go through the grouped GEMM, while Identity and Zero are handled
on the sorted tail (they have no weight GEMM). This is the gap the older GLU-only grouped path could not
close — and it needs no hand-written backward, since `torch._grouped_mm` differentiates itself.

### The win is governed by tokens-per-expert

Grouped beats per-expert exactly when **tokens-per-expert (`routed_tokens / E_glu`) is low** — the
many-experts regime. Measured on an RTX PRO 6000 Blackwell (bf16, forward+backward, vs the per-expert
champion):

| Tokens per expert | Example | Grouped vs per-expert | Memory |
|---|---|---|---|
| ~500 | 64 experts | **~3.0×** | lower than per-expert |
| ~960 | 32 experts | **~2.1–2.6×** | ≤ per-expert |
| ~1800 | 16 experts | **~1.6×** | ≈ per-expert |
| ~3300 | 9 experts (BiBo) | ~1.2× | ≈ per-expert |
| high | few experts, large batch | per-expert wins | per-expert leaner |

Scaling the token count **up** at a fixed expert count actually favors per-expert (its GEMMs grow large
and efficient), so the grouped edge is about expert count and routing sparsity, not batch size. This
applies to **ordinary pure-GLU MoEs too** — a stack with no Identity/Zero is just the case where the
special-expert tail is empty (measured pure-GLU: 32 experts → ~2.6× at lower memory; 16 → ~1.8×).

### Automatic dispatch

You do not choose the path — `moe()` on `sm_120` picks it:

- **Grouped** when `torch._grouped_mm` is available (PyTorch ≥ ~2.5, `sm_80+`), the dtype is bf16/fp16,
  shapes are 16-byte-aligned, **and** tokens-per-expert ≤ `GROUPED_TOKENS_PER_EXPERT_MAX` (2048).
- **Per-expert** otherwise, and as the universal fallback — correct on every shape and arch, and more
  memory-frugal at high `top_k`.

<Aside type="note" title="Speed and memory are a frontier">
Grouped processes all tokens' gradients together (faster, more live memory); per-expert loops
expert-by-expert, freeing as it goes (slower, leaner). The shipped grouped path is the **balanced** point —
at or below per-expert memory for `top_k=2` while 2–3× faster. A gradient-checkpointed (leaner, slower)
and a manual-backward (faster, heavier) variant sit at the ends of that frontier; the dispatch ships the
balanced one. Pick per-expert explicitly if you are memory-bound at high `top_k`.
</Aside>

### Path selection

`moe` auto-dispatches between these backends:

| Function | When | Notes |
|---|---|---|
| `moe` | auto | On `sm_120`: cuBLAS grouped (`moe_grouped_cublas_polyglu`) when supported and tokens-per-expert ≤ 2048; else per-expert. On `sm_75`/older: always per-expert. |
| `moe_grouped_cublas_polyglu` | `sm_120`, many experts | One cuBLAS grouped GEMM (`torch._grouped_mm`) over all routed tokens; PolyGLU + Identity/Zero; autograd-native. The Blackwell scale win. |
| `moe_per_expert` | small token counts, all of Turing, high tokens-per-expert, or memory-bound at high `top_k` | Sorted dispatch + per-expert cuBLAS GEMMs + fused PolyGLU + weighted scatter, manual backward. Correct on every arch; the universal fallback. |
| `moe_grouped` | legacy / parity only | The original GLU-only `tl.dot` grouped GEMM. Catastrophic on Turing, superseded by the cuBLAS path on Blackwell. Not used by `moe()`. |
| `moe_eager` | reference only | The slow masked-loop baseline the fused paths beat — keep for parity checks. |

<Aside type="caution" title="Two grouped implementations — they differ on special experts">
1. **`sm_120` cuBLAS grouped** (`moe_grouped_cublas_polyglu`) — **handles Identity/Zero correctly** and is
   what `moe()` dispatches to on Blackwell when tokens-per-expert is low. No special-expert restriction.
2. **Legacy `tl.dot` grouped** (`moe_grouped`, from `sm75`) — runs a GLU over every expert uniformly, so it
   is correct **only** for pure-GLU stacks (codes `0/1/2`), and is catastrophic on Turing (~0.1× vs compiled
   eager). `moe()` never uses it; it is kept for parity benches.

So on Turing you always get `moe_per_expert`; on Blackwell you get the cuBLAS grouped path in its win regime
and `moe_per_expert` (correct on every arch, a ~4× win itself) everywhere else.
</Aside>

## Usage

```python
import torch
from kernels.sm75 import moe

N, H, I, E, k = 16384, 512, 768, 11, 2
hidden    = torch.randn(N, H, device="cuda", dtype=torch.float16)
gate_up   = (torch.randn(E, 2 * I, H, device="cuda", dtype=torch.float16) * 0.02).requires_grad_(True)
down      = (torch.randn(E, H, I, device="cuda", dtype=torch.float16) * 0.02).requires_grad_(True)
act_codes = torch.tensor([e % 3 for e in range(E)], device="cuda", dtype=torch.int32)  # PolyGLU triples

# idx, weights come from your router:
idx     = torch.randint(0, E, (N, k), device="cuda")
weights = torch.rand(N, k, device="cuda", dtype=torch.float16)

out = moe(hidden, idx, weights, gate_up, down, act_codes)   # (N, H)
out.sum().backward()                                        # grads flow to the experts
```

See [`examples/moe_usage.py`](https://github.com/IsNoobgrammer/triton-kernel-fused/blob/master/examples/moe_usage.py)
for a complete runnable example, including a full MoE layer wired up as an `nn.Module`.
