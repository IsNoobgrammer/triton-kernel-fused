A fused implementation of the **Muon** optimizer step using a Polar-Express Newton-Schulz
orthogonalization. Muon orthogonalizes each 2D (or per-slice 3D) weight's momentum before applying it —
the Newton-Schulz iteration drives the matrix's singular values toward 1. This implementation keeps the
algorithm identical to the established Polar-Express baseline and wins purely by removing launch overhead
and batching same-shape work into larger GEMMs.

## Why it wins

This is **not** a Triton kernel — the step is GEMM-bound (Newton-Schulz is three matmuls per iteration,
which is the algorithmic floor), and a hand-written `tl.dot` Newton-Schulz loses to cuBLAS on Turing. The
wins are all launch-overhead and batched-GEMM levers:

1. **`torch._foreach_*`** collapses the per-parameter momentum / Nesterov sweeps from `N × (several
   launches)` down to a handful.
2. **`baddbmm`** folds each Newton-Schulz axpy (`b·A + c·(A@A)`, then `a·X + B@X`) directly into the cuBLAS
   call — no separate pointwise kernels.
3. **`ns_dtype` selects the compute precision.** On a T4, fp16 engages the tensor cores; fp32 runs on slow
   CUDA cores and bf16 has **no tensor cores at all** on `sm_75`. fp16 is the dominant lever there.
4. **Same-shape parameters batch into one GEMM.** Many small 2D weights and stacked 3D expert weights are
   grouped so the Newton-Schulz runs as one larger, better-utilized matmul.

<Aside type="note">
Capturing the whole step as a CUDA graph was tried and **discarded**: it cut launches by more than half and
erased "Command Buffer Full" from the profile, but wall-clock did not move — the step is GPU-compute-bound,
not launch-bound. The capture path is kept as an opt-in (`use_graph=True`) for a genuinely launch-bound
host, but it is off by default.
</Aside>

## Performance

Measured on a Tesla T4 (`sm_75`), against a `torch.compile`'d Polar-Express Muon baseline, **under a hard
memory gate** (fused peak memory must stay at or below the baseline's peak):

| Parameter count | Speedup | Peak memory |
|---|---|---|
| ~75M (48 tensors) | **~1.09×** | ≤ baseline |
| ~300M (192 tensors) | **~1.03–1.06×** | ≤ baseline |

Parity vs the trusted reference is ~7e-7 (fp32 path). A faster setting exists if you can spend memory — see
[Fast mode](#fast-mode--trading-memory-for-speed).

## API

```python
FusedMuon(
    params,
    lr=0.02,
    momentum=0.95,
    nesterov=True,
    weight_decay=0.0,
    ns_dtype=torch.float16,            # T4 default; fp32 = safe fallback, bf16 = Ampere/Hopper only
    scale_mode="jordan",               # "jordan" | "moonlight"
    ns_batch_elems=4 * 1024 * 1024,    # batched Newton-Schulz size cap (the memory gate)
    use_graph=False,                   # opt-in CUDA-graph capture (off by default)
)
```

<Aside type="caution">
`FusedMuon` only steps **2D and 3D** parameters that have a gradient (3D stacked experts are orthogonalized
per slice). Route 1D parameters (norms, biases), embeddings, and convolution kernels to a standard
optimizer such as AdamW. The conventional Muon setup uses two parameter groups: Muon for the matrices,
AdamW for everything else.
</Aside>

### Key arguments

| Argument | Default | Notes |
|---|---|---|
| `lr` | `0.02` | Learning rate. |
| `momentum` | `0.95` | Momentum coefficient. |
| `nesterov` | `True` | Use Nesterov momentum. |
| `weight_decay` | `0.0` | Decoupled (AdamW-style) weight decay. |
| `ns_dtype` | `torch.float16` | Newton-Schulz compute dtype. **fp16** for T4 tensor cores; **fp32** for a numerically-safe fallback; **bf16** only on Ampere/Hopper. Normalization is always fp32 regardless. |
| `scale_mode` | `"jordan"` | `"jordan"` = `max(1, rows/cols)**0.5` (the Polar-Express baseline). `"moonlight"` = `0.2·sqrt(max(rows, cols))` (a consistent-RMS scale that puts the LR in the AdamW band). |
| `ns_batch_elems` | `4M` | Caps `rows × r × c` per batched Newton-Schulz call — the **memory gate**. The default keeps peak memory below the baseline. |
| `use_graph` | `False` | Capture the step as a CUDA graph and replay it. A wall-clock wash on a compute-bound host; only helps a launch-bound one (slow CPU / many tiny params). Bakes in static `lr`/`wd`/`momentum`. |

### Fast mode — trading memory for speed

`ns_batch_elems` is a frontier knob. Raising it batches more parameters into each Newton-Schulz call, which
runs bigger and better-utilized GEMMs on the T4 — but it costs more transient memory:

| `ns_batch_elems` | Speedup (75M / 300M) | Peak memory |
|---|---|---|
| `4M` (default) | ~1.09× / ~1.05× | ≤ baseline |
| `64M` (fast mode) | **~1.16× / ~1.10×** | exceeds baseline |

The default is gated to stay within the baseline's peak. If you have VRAM headroom over the baseline, set
`ns_batch_elems=64 * 1024 * 1024` for the faster setting:

```python
opt = FusedMuon(matrix_params, lr=0.02, ns_batch_elems=64 * 1024 * 1024)   # fast mode
```

<Aside type="note">
**Per-arch defaults.** The numbers above are `sm75` (T4). On Blackwell the mem-gated knee moves up:
`kernels.sm120.FusedMuon` defaults `ns_batch_elems` to **8M** (measured ~2.42–2.47× at peak ≤ baseline on
both 75M and 302M, vs 4M's ~2.0×). Same class, same API — just the Blackwell-tuned default. The
Newton-Schulz dtype stays **fp16** on every arch: bf16's 7 mantissa bits orthogonalize less precisely than
fp16's 10, and Blackwell runs fp16 on full-rate tensor cores, so fp16 is both tighter and fast.
</Aside>

## Scaling the Newton-Schulz iteration count

The default recipe runs **5** Newton-Schulz iterations. Some recent work argues for more
(around **10**) for tighter orthogonalization. Since each iteration is a fixed three matmuls,
scaling the count is a clean test of whether the fusion advantage is a quirk of 5 iterations
or a property of the step itself. It is the latter — the advantage holds across the range.

Measured on an **RTX PRO 6000 Blackwell (`sm_120`)**, fused fp16 vs a `torch.compile`'d
(`fullgraph`) Polar-Express baseline, both at the **same** iteration count, median step time:

| NS iters | 75M fused / compiled | speedup | 302M fused / compiled | speedup |
|---|---|---|---|---|
| 5 | 7.44 / 16.31 ms | **2.19×** | 28.8 / 64.9 ms | **2.26×** |
| 6 | 8.41 / 18.25 ms | **2.17×** | 31.6 / 72.7 ms | **2.30×** |
| 7 | 9.40 / 19.91 ms | **2.12×** | 35.2 / 90.1 ms | **2.56×** |
| 8 | 10.39 / 21.82 ms | **2.10×** | 37.6 / 86.7 ms | **2.30×** |
| 9 | 11.38 / 23.65 ms | **2.08×** | 41.1 / 94.1 ms | **2.29×** |
| 10 | 12.25 / 26.61 ms | **2.17×** | 44.7 / 103.1 ms | **2.31×** |

The fused-vs-compiled advantage is essentially **flat across the range** — ~2.1× at 75M and
~2.3× at 302M — so the win is a property of the fusion, not of one iteration count. Two
consequences are worth calling out, because they make the higher iteration count cheap to adopt:

- **The marginal cost of each added iteration is ~half.** Going from 5 to 10 iterations costs
  the fused path **~0.96 ms/iter** (75M) and **~3.19 ms/iter** (302M), versus **~2.06 ms/iter**
  and **~7.64 ms/iter** for the compiled baseline — roughly **2.1–2.4× cheaper to add** an
  iteration. (Expressed as throughput retained, both paths keep ~61–64% of their 5-iter speed
  at 10 iterations — the *relative* slope is the same; the fused path's advantage is the
  constant ~2× factor on top, so the absolute time lost per iteration is half.)
- **Doubling the iteration count is still cheaper than the baseline at 5.** Fused at 10
  iterations (12.25 ms / 44.7 ms) beats the compiled baseline at 5 (16.31 ms / 64.9 ms). You can
  move to 10-iteration Newton-Schulz on `FusedMuon` and still come out ahead of `torch.compile`
  at today's default.

Peak memory is **flat across the iteration count** (Newton-Schulz reuses its working buffers),
so more iterations stay within the same memory gate.

<Aside type="note">
This is a runtime-scaling study: step time is set by the iteration **count** (three matmuls
each), and the per-iteration coefficient values only affect orthogonalization quality. The 5
Polar-Express tuples are tuned per iteration; for counts above 5 the settled (last) tuple is
repeated to reach `N`. Pass your own `coeffs` tuple of length `N` to `FusedMuon` /
`newton_schulz` to run `N` iterations.
</Aside>

## Symmetric-matmul Newton-Schulz (Blackwell)

Two of the three Newton-Schulz GEMMs are **symmetric**: `A = X·Xᵀ` and `A·A`. A symmetric product
`M·Mᵀ` only needs its upper triangle — compute those output tiles, transpose-copy each to its mirror,
skip the lower triangle, and you spend about **half** the GEMM FLOPs. On `sm_120` this is the default
Newton-Schulz:

```python
A = symmul(X)              # X·Xᵀ   — Triton triangle + mirror   (~½ FLOPs)
B = symmul_axpy(A, b, c)   # b·A + c·A²   — fused into the kernel epilogue (~½ FLOPs)
X = baddbmm(X, B, X, β=a)  # a·X + B·X — non-symmetric, stays cuBLAS
```

This is **orthogonal** to the launch/batching wins above, so it composes with them: `kernels.sm120`
batches the symmetric kernel across same-shape parameters in one launch. The symmetric matmul is adapted
from [nil0x9/flash-muon](https://github.com/nil0x9/flash-muon) (Triton triangle + transpose-copy; idea
originally by Laker Newhouse et al.), with a batch dimension and a fused polynomial epilogue added.

In short, **`amalg` (amalgamated) = `fused` + `symmul`** — our batched, launch-collapsed optimizer
(`fused`) plus the symmetric-matmul FLOP cut (`symmul`). The two levers are orthogonal, so they stack.

Measured on an **RTX PRO 6000 Blackwell (`sm_120`)**, single square matrix, Newton-Schulz(5), fp16 —
five implementations (`eager` = naive PyTorch, `compiled` = `torch.compile`, `ours-fused` = our
cuBLAS+`baddbmm` NS, `flash` = flash-muon's exact `fast_newtonschulz`, `ours-amalg` = symmetric-matmul):

| dim | compiled | ours-fused | flash | **ours-amalg** | amalg vs compile |
|---|---|---|---|---|---|
| 1024 | 0.249 ms | 0.237 | 0.325 | **0.237** | 1.05× |
| 2048 | 1.076 ms | 1.050 | 0.889 | **0.755** | **1.43×** |
| 4096 | 5.800 ms | 5.502 | 4.381 | **3.847** | **1.51×** |
| 8192 | 44.23 ms | 43.31 | 33.83 | **31.05** | **1.42×** |

![Newton-Schulz step time vs matrix dim, five implementations, log-log](../../../assets/muon/chart_time.png)

![Speedup over torch.compile by matrix dim — ours-amalg wins at every scale](../../../assets/muon/chart_speedup.png)

`ours-amalg` (green) leads at every size ≥2048. `flash` loses at 1024 (kernel launch outweighs the tiny
GEMM) but its symmetric trick pulls ahead of dense cuBLAS by 4096 — and `ours-amalg` stays ahead of
`flash` by batching same-shape work and folding the polynomial into the kernel epilogue.

At the full optimizer step (dense ~0.4B params), the batched state composes with the symmetric cut:

| optimizer | step | peak | speedup |
|---|---|---|---|
| `torch.compile` | 124.6 ms | 3846 MB | 1.00× |
| ours-fused (cuBLAS) | 119.0 ms | 3519 MB | 1.05× |
| flash (nil0x9) | 109.7 ms | 4779 MB | 1.14× |
| **ours-amalg** | **91.2 ms** | 4725 MB | **1.37×** |

![Full Muon optimizer step time, five implementations, dense 0.4B params](../../../assets/muon/chart_opt.png)

The advantage is scale-invariant from 1B to 2.6B params (width *and* depth): ~1.32× over `torch.compile`,
~1.28× over our own champion, at ≤ compiled memory.

<Aside type="note">
**Parity — no precision tradeoff.** The symmetric kernel changes only the *order* of floating-point adds
(triangle+mirror vs full `bmm`), not the math: it runs the identical fp16 NS with fp32 accumulate. Against
an fp32 reference, `ours-amalg` matches `ours-fused` **exactly** — max|Δ| `8.5e-5`–`2.5e-4`, 30–200× under
the Newton-Schulz tolerance (`2e-2`), with identical singular values (SV mean ~0.98). It replaces
`FusedMuon` on Blackwell with no accuracy cost. (flash-muon uses the original Jordan coefficients — SV mean
~0.85 after 5 steps — so it is compared on speed and orthogonality, not bit-parity; our Polar-Express
coefficients reach ~0.98 *and* the faster kernel.)
</Aside>

![Parity: max absolute delta vs fp32 reference per method at 4096, all well under the 2e-2 tolerance](../../../assets/muon/chart_parity.png)

<Aside type="caution">
**The ceiling.** Only 2 of the 3 NS GEMMs are symmetric; `B·X` is a full cuBLAS tensor-core GEMM at
~371 TFLOP/s (near speed-of-light, 46% of the step by `torch.profiler`) and is irreducible by symmetry.
fp8 was tested and rejected — `_scaled_mm` ran *slower* than fp16 here and fp8 shattered orthogonalization
(SV spread `0.001`→`2.2`). So ~1.5× is the algorithmic ceiling for this lever and the kernel sits at
~93–95% of it. Set `use_symmul=False` for the pure-cuBLAS path (no Triton — for a Triton-free environment
or an exact-reference run); the opt-in CUDA-graph path also stays on cuBLAS.
</Aside>

## Usage

```python
import torch
from kernels.sm75 import FusedMuon

# Two groups: matrices -> Muon, everything else -> AdamW.
matrix_params = [p for n, p in model.named_parameters() if p.ndim in (2, 3) and "embed" not in n]
other_params  = [p for n, p in model.named_parameters() if p.ndim < 2 or "embed" in n]

muon  = FusedMuon(matrix_params, lr=0.02, momentum=0.95, weight_decay=0.01)
adamw = torch.optim.AdamW(other_params, lr=3e-4, weight_decay=0.01)

loss = model(batch).loss
loss.backward()
muon.step();  adamw.step()
muon.zero_grad(set_to_none=True);  adamw.zero_grad(set_to_none=True)
```

## Distributed (DDP)

`DistributedMuon` is a drop-in subclass for `DistributedDataParallel`. Each rank computes Newton-Schulz for
a FLOP-balanced subset of the parameters and broadcasts its packed updates; every rank applies the full
update set.

```python
from kernels.sm75 import DistributedMuon

opt = DistributedMuon(matrix_params, lr=0.02, weight_decay=0.01, process_group=None)
```

- **Bit-identical** to a replicated `FusedMuon` — the same all-reduced gradients in produce the same
  weights out; the Newton-Schulz work is relocated, not changed.
- Each rank does ~`1/world_size` of the Newton-Schulz work and stores momentum **only for its owned
  parameters** (less optimizer-state memory).
- Communication is `world_size` broadcasts of packed update blobs — the same total volume as one
  all-gather, on top of DDP's existing gradient all-reduce.

It assumes gradients are already all-reduced across ranks (the DDP default) and that the parameters share
one dtype.

## `newton_schulz` (standalone)

The orthogonalization is exported on its own for direct use:

```python
from kernels.sm75 import newton_schulz

G = torch.randn(512, 512, device="cuda", dtype=torch.float16)
O = newton_schulz(G, ns_dtype=torch.float16)   # singular values driven toward 1
```

It accepts a 2D matrix (unsqueezed to `(1, A, B)` internally) or a 3D stack `(E, A, B)` that batches over
`E`. Normalization is always done in fp32 (an fp16 sum-of-squares of a unit `512²` matrix would overflow);
the iteration GEMMs run in `ns_dtype`.
