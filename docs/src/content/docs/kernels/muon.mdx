A fused implementation of the **Muon** optimizer step using a Polar-Express Newton-Schulz
orthogonalization. Muon orthogonalizes each 2D (or per-slice 3D) weight's momentum before applying it —
the Newton-Schulz iteration drives the matrix's singular values toward 1. This implementation keeps the
algorithm identical to the established Polar-Express baseline and wins purely by removing launch overhead
and batching same-shape work into larger GEMMs.

## Why it wins

This is **not** a Triton kernel — the step is GEMM-bound (Newton-Schulz is three matmuls per iteration,
which is the algorithmic floor), and a hand-written `tl.dot` Newton-Schulz loses to cuBLAS on Turing. The
wins are all launch-overhead and batched-GEMM levers:

1. **`torch._foreach_*`** collapses the per-parameter momentum / Nesterov sweeps from `N × (several
   launches)` down to a handful.
2. **`baddbmm`** folds each Newton-Schulz axpy (`b·A + c·(A@A)`, then `a·X + B@X`) directly into the cuBLAS
   call — no separate pointwise kernels.
3. **`ns_dtype` selects the compute precision.** On a T4, fp16 engages the tensor cores; fp32 runs on slow
   CUDA cores and bf16 has **no tensor cores at all** on `sm_75`. fp16 is the dominant lever there.
4. **Same-shape parameters batch into one GEMM.** Many small 2D weights and stacked 3D expert weights are
   grouped so the Newton-Schulz runs as one larger, better-utilized matmul.

<Aside type="note">
Capturing the whole step as a CUDA graph was tried and **discarded**: it cut launches by more than half and
erased "Command Buffer Full" from the profile, but wall-clock did not move — the step is GPU-compute-bound,
not launch-bound. The capture path is kept as an opt-in (`use_graph=True`) for a genuinely launch-bound
host, but it is off by default.
</Aside>

## Performance

Measured on a Tesla T4 (`sm_75`), against a `torch.compile`'d Polar-Express Muon baseline, **under a hard
memory gate** (fused peak memory must stay at or below the baseline's peak):

| Parameter count | Speedup | Peak memory |
|---|---|---|
| ~75M (48 tensors) | **~1.09×** | ≤ baseline |
| ~300M (192 tensors) | **~1.03–1.06×** | ≤ baseline |

Parity vs the trusted reference is ~7e-7 (fp32 path). A faster setting exists if you can spend memory — see
[Fast mode](#fast-mode--trading-memory-for-speed).

## API

```python
from kernels.sm120 import FusedMuon

FusedMuon(
    params,
    lr=3e-4,                           # AdamW-band: every scale_mode targets update RMS 0.2
    momentum=0.95,
    nesterov=True,
    weight_decay=0.0,
    ns_dtype=torch.float16,            # fp16 (tensor cores); fp32 = safe fallback, bf16 = Ampere/Hopper
    scale_mode="aurora",               # update row shape: "polar" | "normuon" | "aurora" (all AdamW LR band)
    aurora_k=None,                     # aurora polar-solve count; None = default (1)
    ns_batch_elems=8 * 1024 * 1024,    # batched Newton-Schulz size cap (the memory gate); sm120 knee = 8M
    use_symmul=True,                   # MASTER Triton switch. False = pure-cuBLAS champion (no Triton)
    use_gram=True,                     # when Triton is on: Gram-space NS (default) vs symmul NS
    gram_restarts=None,                # override gram restart placement; None = the tuned default ([3])
    use_graph=False,                   # opt-in CUDA-graph capture (off by default; captures the NS too)
)
```

On `sm_120` `FusedMuon` is a single class that dispatches across **three** interchangeable Newton-Schulz
backends — Gram-space (default), symmetric-matmul, and pure cuBLAS — selected by two flags plus per-call
shape gates. There is no separate optimizer to import; the state dict and constructor are identical across
all modes. See [Three Newton-Schulz backends](#three-newton-schulz-backends) for the exact precedence.

<Aside type="caution">
`FusedMuon` only steps **2D and 3D** parameters that have a gradient (3D stacked experts are orthogonalized
per slice). Route 1D parameters (norms, biases), embeddings, and convolution kernels to a standard
optimizer such as AdamW. The conventional Muon setup uses two parameter groups: Muon for the matrices,
AdamW for everything else.
</Aside>

### Key arguments

| Argument | Default | Notes |
|---|---|---|
| `lr` | `3e-4` | Learning rate. All scale modes target update RMS `0.2`, so tune this like an AdamW LR (reuse the one you already have). |
| `momentum` | `0.95` | Momentum coefficient. |
| `nesterov` | `True` | Use Nesterov momentum. |
| `weight_decay` | `0.0` | Decoupled (AdamW-style) weight decay. |
| `ns_dtype` | `torch.float16` | Newton-Schulz compute dtype. **fp16** for T4 tensor cores; **fp32** for a numerically-safe fallback; **bf16** only on Ampere/Hopper. Normalization is always fp32 regardless. |
| `scale_mode` | `"aurora"` | Row shape of the update — see [Update scaling](#update-scaling-scale_mode). All modes target update RMS `0.2`, so **your AdamW LR and weight decay carry over** regardless of mode. Default `"aurora"` additionally fixes the tall-matrix "neuron death" issue. |
| `aurora_k` | `None` | Number of prescale→orthogonalize passes for `scale_mode="aurora"` (`None` = 1). `1` matches the paper's `2` for our matrices at half the cost; `2` = full paper Aurora. Ignored for non-aurora modes. |
| `ns_batch_elems` | `8M` (sm120) | Caps `rows × r × c` per batched Newton-Schulz call — the **memory gate**. `sm120` default is 8M (the Blackwell knee); `sm75` is 4M. |
| `use_symmul` | `True` (sm120) | **Master Triton switch.** `True` runs a Triton Newton-Schulz backend (gram or symmul, per `use_gram`); `False` runs the pure-cuBLAS champion with **no Triton launched at all** (a Triton-free environment or an exact-reference run). When `False`, `use_gram` and `gram_restarts` are ignored. See [Three Newton-Schulz backends](#three-newton-schulz-backends). |
| `use_gram` | `True` (sm120) | When Triton is on (`use_symmul=True`), selects the **Gram-space** NS (default, fastest) over the symmetric-matmul NS. | `gram_restarts` | `None` | Override the gram NS restart placement (an int, a list like `[2, 4]`, or `()` for none). `None` uses the tuned default `[3]`. Only relevant when the gram backend is active; needed mainly for **custom `coeffs`** — find the right value with [`autotune_restarts`](/kernels/muon/gram/#autotune_restarts). |
| `use_graph` | `False` | Capture the step as a CUDA graph and replay it. The captured body runs the active Triton NS too (forced to its eager path, which is graph-capturable). **Speed is a wash** and **memory is a penalty on a realistic (mixed-shape) model** — leave it off unless you've measured a win on your exact model. Bit-identical to eager (verified `0.0` over 6 steps, all three backends). See [The CUDA-graph path](#the-cuda-graph-path-use_graph) for the measured tradeoff. Bakes in static `lr`/`wd`/`momentum`. |

### Three Newton-Schulz backends

`FusedMuon` runs the same Polar-Express Newton-Schulz math through one of three backends. Two flags plus
two per-call shape gates decide which, in this strict order of precedence:

```text
use_symmul=False ────────────────────────────────► pure cuBLAS (no Triton)   [master off-switch]
use_symmul=True
   └─ use_gram=True  (default)
        └─ gram NS, which self-gates per matrix:
              r = max(rows,cols)/min(rows,cols) >= 1.5  AND  min(rows,cols) >= 2048 ─► Gram-space NS
              otherwise ──────────────────────────────────────────────────────────► symmul NS
                 which itself gates: min(rows,cols) >= 2048 ─► symmul, else ────────► cuBLAS
   └─ use_gram=False
        └─ symmul NS  (same dim>=2048 gate to cuBLAS)
```

The upshot: **you never regress by turning a backend on.** Each one falls back to the exact cuBLAS champion
in the regime where it would lose (small or square matrices), so the fast path only engages where it has
been measured to win. The two levers you actually set:

| Goal | Setting |
|---|---|
| Fastest (default) — Gram-space NS where it wins, symmul/cuBLAS elsewhere | `use_symmul=True, use_gram=True` |
| Symmetric-matmul NS only (skip gram) | `use_symmul=True, use_gram=False` |
| Pure cuBLAS champion, no Triton (Triton-free box / exact reference) | `use_symmul=False` |

<Aside type="caution">
`use_gram=True` with `use_symmul=False` does **not** run the gram backend — `use_symmul=False` is the master
off-switch and wins. To use gram you need Triton on (`use_symmul=True`, the default).
</Aside>

### The CUDA-graph path (`use_graph`)

`use_graph=True` captures the whole step as a CUDA graph and replays it, cutting kernel-launch overhead.
It is **off by default** because the Muon step is GPU-compute-bound, not launch-bound — so on a realistic
model the launch saving buys nothing and the graph's static memory pool costs you.

Measured on an **RTX PRO 6000 Blackwell (`sm_120`)**, a transformer stack at `d=4096` (7 matrices/layer:
4 attention `4096²`, 2 MLP `11008×4096`, 1 MLP `4096×11008`), fp16, absolute step-overhead memory (peak
minus resident params + momentum), across all three backends:

| Matrices | Step time (graph vs eager) | Eager overhead | Graph overhead | Graph memory penalty |
|---|---|---|---|---|
| 7 (1 layer) | wash (~1.0×) | 1082 MB | 1126 MB | +44 MB (1.04×) |
| 21 (3 layers) | wash | 1854 MB | 2702 MB | +848 MB (**1.46×**) |
| 56 (8 layers) | wash | 3784 MB | 6562 MB | +2778 MB (**1.73×**) |
| 112 (16 layers) | wash | 6872 MB | 12738 MB | +5866 MB (**1.85×**) |

Two things to take away:

- **Speed is a wash at every size** — graph replay lands within ~1% of eager. There is no throughput win to
  chase on this step.
- **The memory penalty grows with the matrix count.** The graph must hold *all* chunks' transient buffers
  live in its static pool at once (eager frees them between chunks), and because a real model has *mixed*
  shapes the pool cannot amortize across many same-shape tensors. By a 16-layer model the graph nearly
  **doubles** the step's memory overhead.

<Aside type="note">
**When it can still help.** The one regime where the static pool amortizes is a handful of *identical*-shape
matrices and nothing else — there the graph's single pool can undercut eager's repeated allocations. That is
narrow; for any normal transformer (mixed attention + MLP shapes) leave `use_graph=False`. The capture is
**correctness-safe** either way — graph replay is bit-identical to eager (max\|Δ\| `0.0` over 6 steps for
gram, symmul, and cuBLAS alike), including the gram backend's mid-loop restart — so the only reason it is
off by default is the memory tradeoff, not any accuracy risk.
</Aside>

### Fast mode — trading memory for speed

`ns_batch_elems` is a frontier knob. Raising it batches more parameters into each Newton-Schulz call, which
runs bigger and better-utilized GEMMs on the T4 — but it costs more transient memory:

| `ns_batch_elems` | Speedup (75M / 300M) | Peak memory |
|---|---|---|
| `4M` (default) | ~1.09× / ~1.05× | ≤ baseline |
| `64M` (fast mode) | **~1.16× / ~1.10×** | exceeds baseline |

The default is gated to stay within the baseline's peak. If you have VRAM headroom over the baseline, set
`ns_batch_elems=64 * 1024 * 1024` for the faster setting:

```python
opt = FusedMuon(matrix_params, lr=0.02, ns_batch_elems=64 * 1024 * 1024)   # fast mode
```

<Aside type="note">
**Per-arch defaults.** The numbers above are `sm75` (T4). On Blackwell the mem-gated knee moves up:
`kernels.sm120.FusedMuon` defaults `ns_batch_elems` to **8M** (measured ~2.42–2.47× at peak ≤ baseline on
both 75M and 302M, vs 4M's ~2.0×). Same class, same API — just the Blackwell-tuned default. The
Newton-Schulz dtype stays **fp16** on every arch: bf16's 7 mantissa bits orthogonalize less precisely than
fp16's 10, and Blackwell runs fp16 on full-rate tensor cores, so fp16 is both tighter and fast.
</Aside>

## Update scaling (`scale_mode`)

Newton-Schulz returns an (approximately) orthogonal factor whose singular values are all ~1. Before it is
applied as `p += -lr · update`, that factor is scaled. **Every mode targets update RMS `0.2`** — the
Moonlight / DeepSeek-V4 convention — so **your AdamW learning rate and weight decay carry over unchanged,
whatever mode you pick.** One less hyperparameter to tune. The NS iteration and its coefficients are
untouched by `scale_mode`.

What the modes actually differ in is the **row shape** of the update (all in
`kernels/muon/muon_scaling.py`, shared by `sm75` and `sm120`):

| `scale_mode` | What it does | Rows | Orthogonal? |
|---|---|---|---|
| `"polar"` | Plain orthogonalized update `× 0.2·sqrt(max(rows, cols))`. | leverage-skewed on tall matrices | yes |
| `"normuon"` | Per-row EMA second-moment normalize **after** the polar. | uniform | slightly broken by the post-hoc rescale |
| `"aurora"` **(default)** | Divide rows by their norm **before** the polar and re-orthogonalize (`aurora_k` passes). | uniform | yes |

**Why the default is `aurora` (the "neuron death" issue).** On a *tall* weight (rows ≫ cols) the plain
orthogonalized update has row norms that follow the matrix's leverage scores — some neurons get a large
update, others near zero, and the small ones can stall (the Tilde/Aurora finding). Square matrices are fine
(uniform rows), so all three modes coincide there and `polar` is the cheapest. On rectangular weights,
`normuon` fixes the row imbalance cheaply but post-hoc; `aurora` fixes it while keeping the update
orthogonal, at the cost of `aurora_k` polar solves.

Earlier releases shipped extra modes (`moonlight`, `polarexpress`/`jordan`, `unormuon`,
`unormuon_spectral`) that were these same three row shapes under different scalar conventions
(unit spectral norm, spectral gain `k`, ...). Since a scalar is just a fixed LR multiplier, they were
redundant once everything moved to the AdamW band: `moonlight` **is** today's `polar`, and the `unormuon*`
variants collapse into `normuon`. They now raise a `ValueError` naming the replacement.

```python
# default: leverage-aware, orthogonal, AdamW-band — reuse your AdamW LR:
muon = FusedMuon(matrix_params, lr=3e-4)                             # scale_mode="aurora", aurora_k=1

# same LR band, other row shapes:
muon = FusedMuon(matrix_params, lr=3e-4, scale_mode="polar")         # classic Muon direction
muon = FusedMuon(matrix_params, lr=3e-4, scale_mode="normuon")       # cheap per-row fix
```

## Scaling the Newton-Schulz iteration count

The default recipe runs **5** Newton-Schulz iterations. Some recent work argues for more
(around **10**) for tighter orthogonalization. Since each iteration is a fixed three matmuls,
scaling the count is a clean test of whether the fusion advantage is a quirk of 5 iterations
or a property of the step itself. It is the latter — the advantage holds across the range.

Measured on an **RTX PRO 6000 Blackwell (`sm_120`)**, fused fp16 vs a `torch.compile`'d
(`fullgraph`) Polar-Express baseline, both at the **same** iteration count, median step time:

| NS iters | 75M fused / compiled | speedup | 302M fused / compiled | speedup |
|---|---|---|---|---|
| 5 | 7.44 / 16.31 ms | **2.19×** | 28.8 / 64.9 ms | **2.26×** |
| 6 | 8.41 / 18.25 ms | **2.17×** | 31.6 / 72.7 ms | **2.30×** |
| 7 | 9.40 / 19.91 ms | **2.12×** | 35.2 / 90.1 ms | **2.56×** |
| 8 | 10.39 / 21.82 ms | **2.10×** | 37.6 / 86.7 ms | **2.30×** |
| 9 | 11.38 / 23.65 ms | **2.08×** | 41.1 / 94.1 ms | **2.29×** |
| 10 | 12.25 / 26.61 ms | **2.17×** | 44.7 / 103.1 ms | **2.31×** |

The fused-vs-compiled advantage is essentially **flat across the range** — ~2.1× at 75M and
~2.3× at 302M — so the win is a property of the fusion, not of one iteration count. Two
consequences are worth calling out, because they make the higher iteration count cheap to adopt:

- **The marginal cost of each added iteration is ~half.** Going from 5 to 10 iterations costs
  the fused path **~0.96 ms/iter** (75M) and **~3.19 ms/iter** (302M), versus **~2.06 ms/iter**
  and **~7.64 ms/iter** for the compiled baseline — roughly **2.1–2.4× cheaper to add** an
  iteration. (Expressed as throughput retained, both paths keep ~61–64% of their 5-iter speed
  at 10 iterations — the *relative* slope is the same; the fused path's advantage is the
  constant ~2× factor on top, so the absolute time lost per iteration is half.)
- **Doubling the iteration count is still cheaper than the baseline at 5.** Fused at 10
  iterations (12.25 ms / 44.7 ms) beats the compiled baseline at 5 (16.31 ms / 64.9 ms). You can
  move to 10-iteration Newton-Schulz on `FusedMuon` and still come out ahead of `torch.compile`
  at today's default.

Peak memory is **flat across the iteration count** (Newton-Schulz reuses its working buffers),
so more iterations stay within the same memory gate.

<Aside type="note">
This is a runtime-scaling study: step time is set by the iteration **count** (three matmuls
each), and the per-iteration coefficient values only affect orthogonalization quality. The 5
Polar-Express tuples are tuned per iteration; for counts above 5 the settled (last) tuple is
repeated to reach `N`. Pass your own `coeffs` tuple of length `N` to `FusedMuon` /
`newton_schulz` to run `N` iterations.
</Aside>

## The two fast Newton-Schulz backends

Beyond the launch/batching wins above, `FusedMuon` ships two Triton backends that cut the Newton-Schulz
**FLOPs**, each on its own sub-page with full API, gates, and measured numbers:

- **[Symmetric-matmul NS (symmul)](/kernels/muon/symmul/)** — two of the three NS GEMMs (`X·Xᵀ` and `A·A`)
  are symmetric, so compute one triangle and mirror it: about **half** the GEMM FLOPs. Measured ~1.4–1.5×
  over `torch.compile` at the NS level, ~1.37× at the full optimizer step. No precision tradeoff.
- **[Gram-space NS (gram)](/kernels/muon/gram/)** — every NS iterate is a polynomial in `G = X·Xᵀ`, so the
  whole iteration runs on the small `n × n` Gram matrix and the rectangular `B·X` GEMMs collapse into one
  final apply. Measured up to **1.81×** over symmul at the NS level (2.24× batched), **1.22×** at the full
  step, with a Dao-style restart for stability. This is the default (`use_gram=True`).

Both are orthogonal to the launch/batch levers, so they stack, and both self-gate to the exact cuBLAS
champion where they would lose — see [Three Newton-Schulz backends](#three-newton-schulz-backends) for the
dispatch and precedence.

## Usage

```python
import torch
from kernels.sm120 import FusedMuon

# Two groups: matrices -> Muon, everything else -> AdamW.
matrix_params = [p for n, p in model.named_parameters() if p.ndim in (2, 3) and "embed" not in n]
other_params  = [p for n, p in model.named_parameters() if p.ndim < 2 or "embed" in n]

muon  = FusedMuon(matrix_params, lr=0.02, momentum=0.95, weight_decay=0.01)
adamw = torch.optim.AdamW(other_params, lr=3e-4, weight_decay=0.01)

loss = model(batch).loss
loss.backward()
muon.step();  adamw.step()
muon.zero_grad(set_to_none=True);  adamw.zero_grad(set_to_none=True)
```

## Distributed (DDP)

`DistributedMuon` is a drop-in subclass for `DistributedDataParallel`. Each rank computes Newton-Schulz for
a FLOP-balanced subset of the parameters and broadcasts its packed updates; every rank applies the full
update set.

```python
from kernels.sm120 import DistributedMuon

opt = DistributedMuon(matrix_params, lr=0.02, weight_decay=0.01, process_group=None)
```

- **Bit-identical** to a replicated `FusedMuon` — the same all-reduced gradients in produce the same
  weights out; the Newton-Schulz work is relocated, not changed.
- Each rank does ~`1/world_size` of the Newton-Schulz work and stores momentum **only for its owned
  parameters** (less optimizer-state memory).
- Communication is `world_size` broadcasts of packed update blobs — the same total volume as one
  all-gather, on top of DDP's existing gradient all-reduce.

It assumes gradients are already all-reduced across ranks (the DDP default) and that the parameters share
one dtype.

## `newton_schulz` (standalone)

The orthogonalization is exported on its own for direct use. On `sm_120` there are three interchangeable
implementations, all with the same output (to within Newton-Schulz tolerance) and the same core signature:

```python
from kernels.sm120 import newton_schulz, newton_schulz_symmul, newton_schulz_gram

G = torch.randn(2048, 8192, device="cuda", dtype=torch.float16)
O  = newton_schulz(G, ns_dtype=torch.float16)          # pure cuBLAS — the reference champion
Os = newton_schulz_symmul(G, ns_dtype=torch.float16)   # symmetric-matmul — ~1.4-1.5x, parity 8.5e-5
Og = newton_schulz_gram(G, ns_dtype=torch.float16)     # Gram-space — up to ~1.8x over symmul, restart@3
```

All three accept a 2D matrix (unsqueezed to `(1, A, B)` internally) or a 3D stack `(E, A, B)` that batches
over `E`, and share the signature `(G, coeffs=_DSV4_COEFFS, ns_dtype=torch.float16, eps=1e-7)` (gram adds
`gram_dtype`, `restart_at`). Normalization is always done in fp32 (an fp16 sum-of-squares of a unit `512²`
matrix would overflow); the iteration GEMMs run in `ns_dtype`. Each fast variant **self-gates** to the
implementation below it where it would lose, so small matrices never regress:

- `newton_schulz_gram` → `newton_schulz_symmul` when `r < 1.5` or `min(rows, cols) < 2048`
- `newton_schulz_symmul` → `newton_schulz` when `min(rows, cols) < 2048`

For the full API of each fast variant — the lower-level `symmul` / `symmul_axpy` / `symmul2` primitives, the
`GramNewtonSchulz` callable, restart lists, and the `autotune_restarts` tool — see the
[symmul](/kernels/muon/symmul/) and [gram](/kernels/muon/gram/) sub-pages.

### Custom coefficients

The iteration is `X ← a·X + b·(X·Xᵀ)·X + c·(X·Xᵀ)²·X`, with one `(a, b, c)` tuple **per step**. Pass your
own `coeffs` — a tuple of `(a, b, c)` tuples — to use a different recipe. The number of tuples **is** the
number of Newton-Schulz iterations:

```python
# the default: 10 steps of the DeepSeek-V4 hybrid (_DSV4_COEFFS)
from kernels.sm120 import newton_schulz_symmul
O = newton_schulz_symmul(G)

# your own 3-step quintic
my_coeffs = (
    (3.4445, -4.7750, 2.0315),   # step 1
    (3.4445, -4.7750, 2.0315),   # step 2
    (3.4445, -4.7750, 2.0315),   # step 3
)
O = newton_schulz_symmul(G, coeffs=my_coeffs)          # runs 3 iterations
opt = FusedMuon(matrix_params, coeffs=my_coeffs)       # same recipe inside the optimizer
```

The default `_DSV4_COEFFS` is DeepSeek-V4's hybrid schedule (arXiv 2606.19348): 8 steps of the
Keller-Jordan quintic `(3.4445, -4.7750, 2.0315)` — whose large `a` lifts tiny singular values fast —
then 2 polish steps of `(2, -1.5, 0.5)`, whose fixed point pins singular values at exactly 1. It was
chosen over the 8-step Polar-Express schedule (`_PE_COEFFS`, still exported) for **square matrices**:
a square Gaussian update starts with `sigma_min/sigma_max ~ 1/(2n)`, below Polar-Express's `1e-3` design
floor, and the hybrid reaches ~11x better conditioning there (identical on rectangular, at 10 vs 8 steps).

### Scaling the iteration count

Because each tuple is one iteration, you scale the count just by passing a longer `coeffs` — for more
steps, repeat the settled (last) tuple to reach `N`:

```python
from kernels.sm75.muon import _DSV4_COEFFS    # the 10 default tuples

def scaled_coeffs(n_iters: int):
    """Default coeffs scaled to n_iters: truncate, or repeat the last (polish) tuple to extend."""
    if n_iters <= len(_DSV4_COEFFS):
        return _DSV4_COEFFS[:n_iters]
    return _DSV4_COEFFS + (_DSV4_COEFFS[-1],) * (n_iters - len(_DSV4_COEFFS))

opt = FusedMuon(matrix_params, coeffs=scaled_coeffs(12))   # 12-step Newton-Schulz
```

More steps give tighter orthogonalization (SV closer to 1) at a linear cost in matmuls. On Blackwell the
marginal cost per added step is **about half** with the symmetric kernel, and peak memory is flat across the
count (the iteration reuses its buffers) — see [Scaling the Newton-Schulz iteration count](#scaling-the-newton-schulz-iteration-count)
for the measured curve.

## Benchmark

```bash
python bench.py --compile muon       # ~75M parameters
python bench.py --compile muon_big   # ~300M parameters (batched GEMMs dominate)
```
