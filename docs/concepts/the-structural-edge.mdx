The single idea behind this library: **`torch.compile` is already very good, so only write a kernel
where the compiler is structurally unable to do the right thing.**

## Why most custom kernels lose

On a modern GPU, the vast majority of training ops are **memory-bandwidth-bound**, not compute-bound. A
plain elementwise op (a `SiLU`, a residual add, a layernorm) spends nearly all of its time moving bytes
to and from HBM. `torch.compile`'s inductor backend fuses chains of these into a single pass over memory,
which is already at the bandwidth ceiling. A hand-written Triton kernel for the same work reads and writes
the same bytes — so at best it ties, and it usually loses to inductor's tuning.

<Warning>
If your kernel reads and writes the same bytes as the compiled baseline, it will not be faster. Beating
`torch.compile` requires moving **fewer bytes** or doing **less work**, not writing tighter code.
</Warning>

This is why "rewrite everything in Triton" does not pay off, and why this library is deliberately small.

## The four seams worth attacking

A kernel here earns its place only by exploiting an edge the compiler cannot reach:

<AccordionGroup>
  <Accordion title="Data-dependent routing — the shape isn't known until runtime" icon="route">
    In a Mixture-of-Experts layer, which tokens go to which expert is decided at runtime by the router.
    The compiler plans a static graph, so it cannot fuse the dispatch, the per-expert GEMMs, the
    activation, and the weighted scatter into one data-dependent pass. The [`moe`](/kernels/moe) kernel
    does exactly this — and it is the one win here that survives even against a compiled baseline by a
    wide margin (~2.87× on T4).
  </Accordion>
  <Accordion title="Terminal-loss fusion — never materialize the giant intermediate" icon="layer-group">
    A language-model head produces an `(N, V)` logits tensor — at vocab 80k+ and 16k tokens, that single
    intermediate is the memory bottleneck and frequently OOMs. But cross-entropy's gradient needs only the
    logits and the labels, so the [`fused_linear_cross_entropy`](/kernels/cross-entropy) kernel streams
    logits in row-chunks, computes the gradient while each chunk is live, and discards it. The `(N, V)`
    tensor is never stored. The win here is **memory**, which is what lets training happen at all.
  </Accordion>
  <Accordion title="Read-once reductions — the compiler emits two passes" icon="arrows-to-dot">
    Some corrections (like the XSA rejection) require normalizing a tensor and then using it. The natural
    eager expression makes the compiler read `V` twice. The [`fused_xsa`](/kernels/xsa) kernel folds the
    normalization, dot product, scale, and subtraction into one pass that reads `V` once — and broadcasts
    GQA in-kernel so the `repeat_kv` copy never materializes.
  </Accordion>
  <Accordion title="Native-op seams — ops the compiler keeps as library calls" icon="scissors">
    `torch.topk` is a library call inductor must leave as a separate kernel — on T4 it and its gather are
    ~36% of the compiled router forward. The [`fused_router`](/kernels/router) replaces it with a bespoke
    in-register top-k argmax in the epilogue, which is essentially free, and owns the backward glue the
    compiler leaves unfused.
  </Accordion>
</AccordionGroup>

There is a fifth, narrower lever for GEMM-bound steps: **collapse launch overhead and batch same-shape
work into bigger GEMMs.** The [`FusedMuon`](/kernels/muon) optimizer step is GEMM-bound (Newton-Schulz is
three matmuls per iteration — the algorithmic floor), so the win there is not a faster matmul but folding
N per-parameter launches into a handful with `torch._foreach_*` and `baddbmm`, and batching same-shape
expert stacks into one larger, better-utilized GEMM.

## Hard-won lessons

These are the rules the project keeps relearning the expensive way. They are worth internalizing before
you write your own kernel.

<Steps>
  <Step title="Profile to find the seam, don't guess">
    The whole game is locating the immovable library call or the redundant pass, then fusing *around* it.
    Out-computing cuBLAS or cuDNN with a hand-written GEMM almost never works on Turing.
  </Step>
  <Step title="A high launch count is not proof of a launch bound">
    A profiler showing "Command Buffer Full" can mean the CPU submission is simply *overlapped behind* a
    saturated GPU — not that the GPU is idle-stalling. Capturing the Muon step as a CUDA graph cut its
    launches by more than half and erased "Command Buffer Full" from the profile, yet wall-clock did not
    move: the step was compute-bound all along. Confirm a launch bound with the wall-clock delta of a CUDA
    graph, not the launch count.
  </Step>
  <Step title="One GPU class does not predict another">
    Tensor-core availability, SRAM size, occupancy behavior, and `tl.dot` scheduling all differ across
    architectures. `tl.dot` loses to cuBLAS for large GEMMs on Turing but can be competitive on Ampere.
    Use a local machine for **correctness** only; trust the **target GPU** for the performance verdict.
  </Step>
  <Step title="Correctness is the gate, never traded for speed">
    Every kernel is validated for forward-output and gradient parity against the eager reference before
    any speed claim. A faster kernel that fails parity is a bug, not a win.
  </Step>
</Steps>
